# 代码提交规范

## 1. 目的

本文档定义本项目后续代码提交的统一规范，目标是：

1. 让每次提交都能快速看懂“改了什么”和“为什么改”。
2. 让版本历史适合作为长期迭代 base 使用。
3. 降低后续排查问题、回滚代码、生成发布说明的成本。

## 2. 总体原则

后续所有提交遵循以下原则：

1. 一次提交只做一类事情。
2. 提交信息必须能独立说明改动目的。
3. 不提交无关改动。
4. 不把临时调试、无意义格式化、杂项修改混在一个提交里。
5. 代码提交前至少保证本地能通过最基本验证。

## 3. 提交信息格式

本项目统一采用 `Conventional Commits` 风格。

标准格式：

```text
<type>(<scope>): <subject>
```

如果不需要 `scope`，可简化为：

```text
<type>: <subject>
```

示例：

```text
feat(positioning): add robust wifi fingerprint scoring
fix(scanner): handle wifi scan throttling on android 13
docs(design): add commit convention document
refactor(ui): split map rendering from activity
test(positioning): add stable observation unit test
build(android): upgrade gradle wrapper to 8.8
```

## 4. type 定义

### `feat`

新增功能时使用。

示例：

```text
feat(positioning): add kalman-based trajectory smoothing
```

### `fix`

修复 bug 时使用。

示例：

```text
fix(scanner): avoid crash when wifi is disabled
```

### `refactor`

重构代码但不改变外部功能时使用。

示例：

```text
refactor(storage): extract fingerprint persistence logic
```

### `docs`

文档新增或文档更新时使用。

示例：

```text
docs(design): update software design document
```

### `test`

新增或修改测试时使用。

示例：

```text
test(positioning): cover outlier suppression case
```

### `build`

构建系统、依赖版本、Gradle、SDK、CI 相关改动时使用。

示例：

```text
build(android): upgrade agp to 8.5.2
```

### `chore`

杂项维护工作，但不属于功能、修复、重构、测试、文档时使用。

示例：

```text
chore(repo): update gitignore rules
```

### `perf`

性能优化时使用。

示例：

```text
perf(positioning): reduce scan window aggregation overhead
```

## 5. scope 建议

`scope` 用来说明改动影响的模块。建议优先使用以下范围：

1. `positioning`
2. `scanner`
3. `storage`
4. `ui`
5. `map`
6. `design`
7. `memory`
8. `android`
9. `repo`
10. `docs`

示例：

```text
feat(storage): support saving collected fingerprint points
fix(ui): correct candidate list refresh timing
docs(memory): add project snapshot template
```

## 6. subject 编写要求

`subject` 编写规则：

1. 使用英文，便于和常见开源规范保持一致。
2. 用简洁动词开头。
3. 不要首字母大写。
4. 不要以句号结尾。
5. 长度尽量控制在 50 到 72 个字符内。

推荐写法：

```text
fix(positioning): reject low-overlap fingerprint matches
```

不推荐写法：

```text
fix: 修改了一些问题
update code
feat: Do many changes.
```

## 7. 提交粒度要求

### 应该拆分提交的情况

以下情况应该拆分成多个 commit：

1. 功能开发和文档更新同时发生。
2. 构建链升级和业务逻辑修改同时发生。
3. UI 调整和定位算法修改同时发生。
4. 重构和 bug 修复同时发生。

示例：

1. `feat(positioning): add candidate confidence radius`
2. `test(positioning): add confidence radius test`
3. `docs(design): document confidence radius model`

### 不应该拆分过细的情况

以下情况不要拆得太碎：

1. 同一功能的实现和必要的小范围配套修改。
2. 一个 bug 修复中必须联动的几个文件。

目标不是提交越多越好，而是每个提交都有清晰边界。

## 8. 提交前检查

每次提交前至少执行以下检查：

1. `git status`
2. 确认没有把无关文件混进来
3. 确认本次提交主题单一
4. 运行最低限度验证

针对本项目，最低建议检查：

```bash
cd /home/nio/workspace/wifi_location/wifi-positioning-app
JAVA_HOME=/home/nio/tools/android-studio/jbr GRADLE_USER_HOME=/home/nio/workspace/wifi_location/.gradle ./gradlew assembleDebug
JAVA_HOME=/home/nio/tools/android-studio/jbr GRADLE_USER_HOME=/home/nio/workspace/wifi_location/.gradle ./gradlew testDebugUnitTest
```

如果只是文档修改，可不跑完整构建，但至少确认改动范围纯文档。

## 9. 推荐提交流程

推荐按以下流程操作：

1. 开发完成后先看变更范围

```bash
git status
git diff
```

2. 只暂存本次要提交的文件

```bash
git add <file1> <file2>
```

3. 再次确认暂存区内容

```bash
git diff --cached
```

4. 使用标准格式提交

```bash
git commit -m "feat(positioning): add robust fingerprint weighting"
```

## 10. 分支建议

如果后续开始持续迭代，建议采用以下分支命名：

1. `main`
   永远保持可作为稳定 base 的主分支。
2. `feature/<topic>`
   新功能分支。
3. `fix/<topic>`
   bug 修复分支。
4. `docs/<topic>`
   文档分支。

示例：

```text
feature/fingerprint-collector
feature/imu-fusion
fix/wifi-scan-timeout
docs/commit-convention
```

## 11. PR 标题建议

如果后续使用 Pull Request，PR 标题建议与主提交类型保持一致，也尽量沿用同样格式：

```text
feat(positioning): add imu-assisted smoothing
fix(storage): prevent duplicate fingerprint point ids
docs(design): refine software design document
```

PR 描述建议包含：

1. 改动目的
2. 主要改动点
3. 验证方式
4. 风险和影响范围

## 12. 本项目特别要求

针对当前 WiFi Location 项目，额外要求如下：

1. 涉及定位算法修改时，提交说明必须明确：
   是为了解决什么问题，例如漂移、抖动、低重叠率误匹配。
2. 涉及扫描逻辑修改时，提交说明必须明确：
   是否影响 Android 版本兼容性、权限、限流行为。
3. 涉及构建链修改时，提交说明必须明确：
   修改了哪个版本，例如 AGP、Gradle、SDK、JDK 依赖。
4. 涉及文档修改时，提交说明必须明确：
   是设计变更、开发思路补充，还是项目快照更新。

## 13. 推荐示例

以下是本项目后续比较推荐直接使用的提交示例：

```text
feat(positioning): add fingerprint collection mode
feat(ui): add factory floor map background support
fix(positioning): reject unstable candidate jumps
fix(scanner): retry scan after throttling
refactor(positioning): separate scoring and filtering stages
perf(positioning): reduce stable observation recomputation
build(android): align gradle and agp versions
test(scanner): add wifi scan result parsing test
docs(design): add code commit convention
chore(repo): ignore android local build artifacts
```

## 14. 禁止事项

后续提交时，尽量避免以下写法：

1. `update`
2. `fix bug`
3. `modify code`
4. `第一次提交`
5. `latest`
6. `提交一下`

这些提交信息无法表达真实变更内容，后期几乎没有检索价值。

## 15. 结论

从现在开始，本项目统一采用：

```text
<type>(<scope>): <subject>
```

推荐后续默认优先使用：

1. `feat`
2. `fix`
3. `refactor`
4. `docs`
5. `test`
6. `build`

如果没有特别原因，后续所有提交都按这个规范执行。
