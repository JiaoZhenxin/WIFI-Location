# Factory WiFi Locator 开发思路

本文档不是正式设计说明，而是给后续开发者或其他 AI 工具接力用的工程笔记，重点说明我为什么这样拆、接下来最值得继续补什么。

## 1. 我对原 demo 的判断

原 demo 可以看作一个 WiFi 指纹定位实验工程，但它有几个不适合直接拿去工厂落地的问题：

1. 业务逻辑太杂。
   登录、网络、调试 UI 和定位逻辑耦合在一起，不适合后续维护。
2. 抗漂移能力不够。
   更像“算一次就出点”，没有真正的轨迹约束。
3. 数据格式不够工程化。
   虽然有指纹数据和参考点坐标，但不够方便现场增量采集和版本管理。

所以我没有在 demo 上硬改，而是新建了一个简化且可扩展的 Android 工程。

## 2. 这次实现的核心思路

我把定位问题拆成四层：

1. 扫描层。
   负责稳定拿到 WiFi 扫描结果。
2. 观测层。
   把最近几帧扫描合成为一个稳态 RSSI 向量。
3. 匹配层。
   把稳态观测和指纹库逐点比较，输出候选点与分数。
4. 轨迹层。
   用时序滤波拒绝异常跳点，让轨迹连续。

漂移问题本质上主要发生在第 2 到第 4 层，所以改进重点也放在这里。

## 3. 我为什么用了这几个算法

### 3.1 中位数 + MAD

工厂 RSSI 经常会突然掉很多或者突然强很多。平均值对异常点不够稳，所以先用中位数定位中心，再用 MAD 判断离群值。这样成本低、实现简单、效果直接。

### 3.2 标准化残差而不是裸 RSSI 差

不同 AP 的稳定性不同。某些 AP 在同一点波动很小，某些波动很大。如果直接拿 `|rssi_live - rssi_fp|` 做距离，稳定 AP 和不稳定 AP 的地位被混为一谈。标准化残差本质上是在做按方差归一化，更合理。

### 3.3 overlap 约束

很多误匹配不是因为距离最小，而是因为只重叠了很少几个 AP。所以我把 overlap ratio 直接写进得分函数，还设了下限。

### 3.4 top-k 质心而不是 winner-takes-all

如果只取第一名，定位会特别容易在相邻参考点之间闪烁。top-k 加权质心会让结果更平滑，也更适合后续时序滤波。

### 3.5 卡尔曼 + 马氏距离门限

单纯做滑动平均会有滞后，而且难处理异常跃迁。卡尔曼的好处是可以显式建模位置和速度，再通过 innovation gating 拒绝不合理观测。对于工厂定位，这是低成本但很有效的一步。

## 4. 当前工程结构的意图

### 4.1 `MainActivity`

尽量只做编排，不做复杂算法。后续如果要改成 MVVM，可以把这部分迁到 `ViewModel`。

### 4.2 `WifiScanner`

我把 Android WiFi 扫描细节单独封装起来，后续如果你想接：

1. `WifiNetworkSuggestion`
2. 企业专用扫描 SDK
3. BLE 扫描
4. RTT

都可以在这层扩展。

### 4.3 `PositioningEngine`

这里就是当前定位内核。后续你可以在不动 UI 的情况下替换为：

1. 随机森林回归
2. XGBoost
3. 深度学习指纹编码器
4. 图优化
5. WiFi + IMU 融合

### 4.4 `TemporalPositionFilter`

这层就是“防漂移缓冲器”。后续若要升级到真正可商用的轨迹约束，可以换成：

1. EKF / UKF
2. 粒子滤波
3. HMM 地图匹配
4. 因子图

## 5. 我刻意保留的扩展点

1. 指纹库存储用 JSON，不用 Room。
   原因是前期调试、导入导出、AI 生成和脚本处理都更方便。
2. 地图先用 `FloorMapView` 手绘，不依赖图片。
   这样最轻、最稳，也方便后续把坐标体系先跑通。
3. 采集功能是“基于当前窗口保存”。
   这样不和扫描节奏强耦合，后续你可以升级成专门的采集流程。

## 6. 如果后面要把项目做强，优先级建议

### 第一优先级

1. 做真正的现场指纹采集流程。
   需要引导用户在每个参考点采 20 到 30 帧，并支持覆盖/合并历史数据。
2. 增加地图底图和区域边界。
   现在只有抽象坐标平面，还没做真实厂区底图和禁行区域约束。
3. 做指纹库管理。
   包括导入、导出、删除点位、更新点位。

### 第二优先级

1. AP 稳定性评估。
   对每个 AP 统计长期方差，自动剔除不稳定热点。
2. 设备型号归一化。
   不同手机的 RSSI 偏置会明显影响定位效果，需要做 per-device calibration。
3. 置信度驱动的 UI。
   当置信度很差时，不应该硬输出点位，而应提示“定位不可靠”。

### 第三优先级

1. IMU 融合。
2. RTT 融合。
3. 多楼层支持。
4. 在线学习和后台模型更新。

## 7. 如果其他 AI 接手，建议它先看什么

按这个顺序看：

1. `MainActivity.java`
2. `WifiScanner.java`
3. `PositioningEngine.java`
4. `TemporalPositionFilter.java`
5. `fingerprint_seed.json`
6. `软件设计文档.md`

这样最容易快速建立上下文。

## 8. 我认为当前版本最真实的边界

这版代码已经能作为一个新项目的起点，也把解决漂移的主路径实现出来了，但还不是“拿到工厂就必然 1 到 2 米”的最终版。真正决定精度上限的仍然是：

1. 参考点密度
2. AP 布局
3. 现场标定质量
4. 手机型号一致性
5. 是否引入 IMU 或 RTT

所以后续开发不要把精度问题完全归因到算法参数，而要把“采集体系”和“部署约束”一起设计进去。
