# Factory WiFi Locator 软件设计文档 v1.1.0

## 版本信息

1. 版本号：`v1.1.0`
2. 变更日期：`2026-05-09`
3. 变更类型：架构职责调整
4. 上一基线：`desgin/软件设计文档.md`

## 变更摘要

1. 将 Android 工程代码结构按 MVC 进行分层。
2. 新增包级职责说明，约束控制器、模型、视图的放置边界。
3. 保持现有业务行为不变，重点解决根包扁平堆放导致的长期维护风险。

## 与上一版差异

1. 新增 `controller / model / view` 三层目录边界。
2. `MainActivity` 从“控制器+业务实现混合体”收敛为控制器入口。
3. 指纹库访问、扫描采集、定位流程编排被拆分到 `model` 下不同子层。

## 4.1 MVC 模块划分

当前工程按 MVC 拆分为以下包层次：

1. `com.nio.wifilocation.controller`
   放控制器，当前由 `MainActivity` 负责权限申请、UI 事件绑定、后台任务调度和界面渲染。
2. `com.nio.wifilocation.model.entity`
   放纯数据对象，包括 `AccessPointStats`、`FingerprintPoint`、`FingerprintDatabase`、`PositionEstimate`。
3. `com.nio.wifilocation.model.repository`
   放数据持久化访问层，当前由 `FingerprintStore` 负责从 assets 和应用私有存储读写指纹库。
4. `com.nio.wifilocation.model.scanner`
   放外部数据采集适配层，当前由 `WifiScanner` 负责 WiFi 扫描循环、广播接收和扫描结果摘要。
5. `com.nio.wifilocation.model.service`
   放业务规则和流程编排，当前包括 `PositioningEngine`、`TemporalPositionFilter`、`PositioningSessionManager`。
6. `com.nio.wifilocation.view`
   放自定义视图和界面展示组件，当前由 `FloorMapView` 负责平面图绘制。

## MVC 放置规则

1. `Activity` 和 `Fragment` 默认只放在 `controller`，不直接持有复杂算法和持久化细节。
2. 算法、会话状态、采集窗口、定位流程等业务逻辑默认进入 `model.service`。
3. 文件存储、数据库、接口访问等数据来源默认进入 `model.repository` 或 `model.scanner`。
4. 自定义 View、绘图组件、纯展示组件默认进入 `view`。
5. 新增代码优先放到已有层级下，避免再次回到根包扁平堆放。
