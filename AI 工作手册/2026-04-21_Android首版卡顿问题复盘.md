# 2026-04-21 Android 首版卡顿问题复盘

## 1. 问题结论

第一次实现存在明显工程失误。

问题不是“功能复杂后才暴露出的细节”，而是一个很基础的 Android 性能问题：

1. 把耗时计算放进了 UI 相关执行路径。
2. 把文件 IO 放进了按钮点击后的同步流程。
3. 在首版交付前没有把“主线程是否阻塞”当成强制检查项。

这类问题在 Android 上属于首轮实现就应该主动规避的内容，没有提前处理好，原因在于实现时过度关注“功能链路先跑通”，低估了交互线程模型的约束。

## 2. 具体错误在哪里

首版实现中的问题主要集中在 [MainActivity.java](/home/nio/workspace/wifi_location/wifi-positioning-app/app/src/main/java/com/nio/wifilocation/MainActivity.java:25)：

1. 扫描结果到达后，直接在相关路径里做了 RSSI 聚合和定位计算。
2. `loadDatabase()` 在主线程上读本地指纹库。
3. `resetDatabase()` 在主线程上做文件恢复和重载。
4. `saveCurrentFingerprint()` 在按钮点击后走了同步数据整理和持久化写入。

这些操作单独看都不算特别重，但在手机上只要叠加：

1. WiFi 扫描回调
2. 文本刷新
3. 地图重绘
4. JSON 读写
5. 指纹评分计算

就足以造成按钮点击发卡、界面停顿、交互不跟手。

## 3. 为什么第一次会犯这个错

根因不是“不懂 Android”，而是执行优先级判断出了偏差。

当时的错误思路是：

1. 先把 WiFi 定位完整链路实现出来。
2. 先确保能扫描、能出点、能保存、能构建。
3. 认为后续再做性能优化也可以。

这个判断在服务端工具、脚本型程序里有时成立，但在 Android 前台交互应用里不成立。原因是：

1. Android 用户第一感知就是按钮是否立刻响应。
2. 一旦主线程被阻塞，软件会被直接判断为“没法用”。
3. 交互流畅性不是二阶段优化项，而是首版可用性的最低门槛。

所以这次不是“功能完成后再优化”的合理取舍，而是把一个本应前置处理的约束后置了。

## 4. 这次修复做了什么

本次修复已经完成以下处理：

1. 把指纹库加载移到后台线程。
2. 把扫描结果聚合和定位计算移到后台线程。
3. 把保存指纹点移到后台线程。
4. 把恢复示例指纹库移到后台线程。
5. 增加扫描窗口并发保护，避免后台任务与 UI 触发并发访问。
6. 在耗时操作期间禁用相关按钮，避免重复点击叠加卡顿。

修复后的主要代码仍在：

1. `assembleDebug`

## 5. 这次暴露出的 AI 执行失误

后续 AI 需要明确避免以下错误模式：

### 错误模式 1

只验证“功能正确”，不验证“交互可用”。

表现：

1. 能构建
2. 能点击
3. 能出结果
4. 但没有检查点击后是否卡顿

### 错误模式 2

默认把 Activity 当成普通协调层，但实际把计算和 IO 都写进去了。

表现：

1. `Activity` 变成了同步大杂烩
2. UI 控件、扫描逻辑、算法逻辑、持久化逻辑直接串在一起

### 错误模式 3

误把“性能问题”归类成后续优化，而不是首版可用性要求。

表现：

1. 先交付一个“理论上能跑”的版本
2. 再等用户反馈卡顿

这会明显降低交付质量。

## 6. 后续 AI 必须遵守的工程约束

今后只要是 Android 前台应用，必须默认执行以下约束。

### 6.1 主线程禁区

以下内容默认不能放在主线程执行：

1. 文件读写
2. JSON 解析和序列化
3. 扫描结果全量聚合
4. 定位算法计算
5. 大量字符串拼接和列表排序
6. 数据库存取

如果确实需要在主线程触发，只能做：

1. 参数采集
2. 按钮状态切换
3. UI 更新
4. 把任务投递到后台

### 6.2 首版交付前强制检查

后续 AI 在 Android 首版交付前，必须至少问自己这几个问题：

1. 点击按钮后是否有同步 IO？
2. 点击按钮后是否有同步排序、聚合、匹配、建模？
3. 是否把回调里的计算直接放在 UI 路径？
4. 是否存在每次扫描都全量刷新大量文本和视图？
5. 是否为重复点击做了状态保护？

如果任一答案是“有”，就不能视为实现完成。

### 6.3 性能优先级规则

对 Android 交互应用，优先级应是：

1. 不阻塞主线程
2. 可用
3. 正确
4. 再扩展功能

不是：

1. 功能先写全
2. 以后再优化体验

## 7. 本项目后续还要继续检查的点

虽然这轮修复已经解决了最明显的卡顿来源，但后续仍建议继续检查：

1. 扫描结果文本刷新频率是否过高
2. 地图重绘是否过于频繁
3. 扫描回调频率和计算频率是否应该解耦
4. 是否需要对候选点文本做降采样刷新
5. 是否需要把定位计算进一步抽到专门组件中

## 8. 给后续 AI 的直接指令

后续如果再接手这个项目，请默认执行以下原则：

1. 先看 `Activity` 里是否有同步计算和同步 IO。
2. 只要是按钮点击或扫描回调路径，先排查主线程阻塞。
3. Android 首版不是“功能跑通就行”，必须同时保证基本流畅。
4. 如果不能现场跑机验证，也要从代码结构上主动避免 UI 线程重活。
5. 不要等用户指出“很卡”之后才回头修这种基础问题。

## 9. 结论

这次问题本质上是一次基础工程纪律失守：首版实现时没有把 Android 主线程约束当成硬边界。后续 AI 不应重复这种错误。对于带前台交互的移动端应用，流畅性不是补充项，而是交付定义的一部分。
