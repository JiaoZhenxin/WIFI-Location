# AI 开发红线清单

本文档是给后续 AI 使用的快速检查清单。目标不是解释原理，而是防止再次犯基础工程错误。

## 1. Android 前台交互红线

以下内容默认禁止直接放在主线程执行：

1. 文件读写
2. JSON 解析和序列化
3. 数据库存取
4. WiFi 扫描结果聚合
5. 定位算法计算
6. 大量排序、过滤、拼接文本
7. 大量列表刷新和高频重绘准备逻辑

只要出现在：

1. 按钮点击回调
2. 扫描结果回调
3. Activity 生命周期回调

就必须优先怀疑是否会卡 UI。

## 2. 首版实现红线

不允许把下面这些问题留到“后续优化”：

1. 点按钮明显卡顿
2. 页面启动明显卡顿
3. 扫描开始后界面掉帧
4. 保存操作阻塞界面
5. 重置操作阻塞界面

对于 Android 前台 app，这些不是优化项，是可用性底线。

## 3. 结构设计红线

不允许把以下职责长期混在 `Activity` 里：

1. UI 控制
2. 持久化读写
3. 算法计算
4. 扫描调度
5. 数据聚合

如果 `Activity` 同时做了上面多项，默认说明实现质量有问题。

## 4. 按钮与交互红线

只要一个操作满足任一条件，就必须做状态保护：

1. 会触发后台任务
2. 会写文件
3. 会改数据库或本地存储
4. 会发起扫描
5. 会执行计算

至少要做到：

1. 操作期间禁用相关按钮
2. 状态文本可见
3. 完成后恢复按钮
4. 失败时给出反馈

## 5. 回调处理红线

高频回调里默认不能直接做重活，尤其是：

1. WiFi 扫描回调
2. 传感器回调
3. 网络回调
4. 定时器回调

高频回调只允许做：

1. 快速收集数据
2. 投递后台任务
3. 轻量状态更新

## 6. 数据一致性红线

后台线程引入后，默认必须检查：

1. 是否有共享集合并发访问
2. 是否有 UI 和后台同时读写同一数据
3. 是否有旧任务结果覆盖新任务结果
4. 是否需要加锁、快照或版本号

如果没有检查这些，说明实现还没闭环。

## 7. 性能检查红线

每次 Android 版本交付前，AI 必须至少检查这 7 个问题：

1. 有无同步 IO 在 UI 路径
2. 有无同步计算在 UI 路径
3. 有无高频回调直接做重活
4. 有无重复点击导致叠加任务
5. 有无无节制的 UI 刷新
6. 有无后台任务结果乱序覆盖
7. 有无只验证“能跑”，没验证“流畅”

只要任一项答案是“有”或“不确定”，就不能认为任务完成。

## 8. 提交前红线

提交代码前，不允许跳过以下检查：

1. `git diff`
2. 确认改动范围干净
3. 构建通过
4. 核心测试通过
5. 明确这次改动有没有影响交互流畅性

## 9. 文档红线

出现以下情况必须补文档：

1. 构建方式变更
2. 线程模型变更
3. 核心算法逻辑变更
4. 数据目录或文档目录调整
5. 新增后续 AI 需要遵守的工程规则

## 10. 当前项目特别红线

对于本项目 `WIFI-Location`，后续 AI 额外必须遵守：

1. 不要把定位计算重新搬回主线程。
2. 不要在保存指纹、恢复指纹库时走同步 UI 路径。
3. 不要在扫描回调里直接叠加大量字符串和重绘逻辑。
4. 修改定位算法时，必须同时考虑流畅性，不只看精度。
5. 修改扫描逻辑时，必须考虑 Android 限流和前台响应。

## 11. 一句话原则

对 Android 前台应用：

**先保证不阻塞，再谈功能完整；先保证可用，再谈后续优化。**
